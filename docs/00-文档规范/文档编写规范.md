## 文档结构规范

### 必需元素

每个文档必须包含以下元素：

1. **标题**：清晰描述文档主题
2. **最后更新时间**：格式为 `**最后更新时间：** YYYY-MM-DD`
3. **目录**（可选）：对于长文档（超过 3 个主要章节）
4. **正文内容**：按逻辑组织的章节
5. **相关文档**（可选）：链接到相关文档

### 文档头部格式

```markdown
# 文档标题

**最后更新时间：** YYYY-MM-DD

## 目录

1. [章节1](#章节1)
2. [章节2](#章节2)
...

---

## 章节1

内容...
```

## 文档命名规范

### 命名原则

1. **使用中文名称**：便于查找和理解
2. **描述性命名**：名称应清晰反映文档内容
3. **避免特殊字符**：不使用空格、特殊符号
4. **使用连字符**：多个词之间使用连字符连接（如需要）

### 命名示例

✅ **好的命名：**
- `软删除完整指南.md`
- `用户管理API文档.md`
- `前端项目开发指南.md`

❌ **不好的命名：**
- `soft-delete.md`（使用英文）
- `文档1.md`（不描述性）
- `用户 管理 API.md`（包含空格）

## 文档内容规范

### 1. 标题层级

- **一级标题（#）**：文档标题，每个文档只有一个
- **二级标题（##）**：主要章节
- **三级标题（###）**：子章节
- **四级标题（####）**：详细说明
- **不建议使用五级及以上标题**

### 2. 代码块

使用语法高亮的代码块：

````markdown
```go
func Example() {
    // Go 代码
}
```

```typescript
const example = () => {
    // TypeScript 代码
}
```

```bash
# Shell 命令
npm install
```
````

### 3. 列表

**无序列表：**
```markdown
- 项目 1
- 项目 2
  - 子项目 2.1
  - 子项目 2.2
```

**有序列表：**
```markdown
1. 第一步
2. 第二步
3. 第三步
```

### 4. 表格

```markdown
| 列1 | 列2 | 列3 |
|-----|-----|-----|
| 数据1 | 数据2 | 数据3 |
| 数据4 | 数据5 | 数据6 |
```

### 5. 强调和标注

- **粗体**：`**重要内容**`
- *斜体*：`*强调内容*`
- `代码`：`` `代码片段` ``
- ✅ 成功标记
- ❌ 失败标记
- ⚠️ 警告标记
- 💡 提示标记

### 6. 链接

**内部链接：**
```markdown
[相关文档](./其他文档.md)
[章节链接](#章节名称)
```

**外部链接：**
```markdown
[AWS 文档](https://docs.aws.amazon.com)
```

## 文档维护规范

### 1. 更新时间

- 每次修改文档时，必须更新文档头部的"最后更新时间"
- 格式：`**最后更新时间：** YYYY-MM-DD`

### 2. 版本控制

- 所有文档使用 Git 进行版本控制
- 重要文档的重大变更应在 commit 信息中说明
- 不保留文档内的历史版本记录

### 3. 文档审查

- 新增文档应经过审查
- 重要文档的修改应经过审查
- 定期审查文档与源码的一致性

### 4. 废弃文档

- 不删除废弃文档，而是在文档开头添加废弃标记：

```markdown
# 文档标题

> ⚠️ **已废弃**：此文档已被 [新文档名称](./新文档.md) 替代，请参考新文档。

**最后更新时间：** YYYY-MM-DD
```

### 5. 文档清理

- 每季度审查一次文档结构
- 删除空文件和重复文档
- 合并内容相似的文档
- 更新文档索引（README.md）

### API 文档模板

```markdown
# API 名称

**最后更新时间：** YYYY-MM-DD

## 接口概述

简要描述 API 的作用。

---

## 接口详情

### 请求

**HTTP 方法：** GET/POST/PUT/DELETE

**请求路径：** `/api/v1/resource`

**请求参数：**

| 参数名 | 类型 | 必填 | 说明 |
|--------|------|------|------|
| param1 | string | 是 | 参数说明 |
| param2 | number | 否 | 参数说明 |

**请求示例：**

\`\`\`json
{
  "param1": "value1",
  "param2": 123
}
\`\`\`

### 响应

**响应格式：**

\`\`\`json
{
  "code": 200,
  "msg": "success",
  "data": {
    "field1": "value1",
    "field2": "value2"
  }
}
\`\`\`

**响应字段说明：**

| 字段名 | 类型 | 说明 |
|--------|------|------|
| code | number | 状态码 |
| msg | string | 消息 |
| data | object | 数据 |

---

## 错误码

| 错误码 | 说明 |
|--------|------|
| 400 | 请求参数错误 |
| 401 | 未授权 |
| 404 | 资源不存在 |
| 500 | 服务器错误 |

---

## 使用示例

### cURL 示例

\`\`\`bash
curl -X POST https://api.example.com/api/v1/resource \
  -H "Content-Type: application/json" \
  -d '{"param1": "value1"}'
\`\`\`

### JavaScript 示例

\`\`\`javascript
const response = await fetch('/api/v1/resource', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json'
  },
  body: JSON.stringify({
    param1: 'value1'
  })
});
\`\`\`
```

## 总结

遵循本规范可以确保：

1. **一致性**：所有文档遵循统一的格式和结构
2. **可维护性**：文档易于更新和维护
3. **可读性**：文档易于阅读和理解
4. **可追溯性**：通过更新时间和 Git 历史追溯变更
5. **专业性**：提升项目文档的整体质量
